# **High‑Level Implementation Plan: Canonical YAML Blueprint System**

## **Phase 0 — Foundations (Prereqs & Alignment)**  
**Goal:** Ensure the system is ready for canonical YAML ingestion and derived‑artifact generation.

### Deliverables  
- **Blueprint canonical model** (C#):  
  - `Blueprint`  
  - `PlacementBlock`  
  - `BlockState`  
  - `BlueprintMetadata`  
  - `BuildPhase`  
- **Decision:** YAML is canonical; everything else is derived.  
- **Decision:** All documentation is autogenerated; hand editing is fallback only.

### Why this matters  
This phase locks the conceptual model so all future phases build on a stable foundation.

---

# **Phase 1 — Canonical YAML Schema & Parser**

## 1.1 Define the YAML schema  
The canonical schema includes:

- Header (id, name, version, dimensions)
- Canonical block list (pos, id, state, tags)
- Optional metadata (author, category, difficulty)
- Optional phases (if user overrides autogen)

No layers, no materials, no legend — those are derived.

## 1.2 Implement YAML parser  
- Use `YamlDotNet`  
- Map YAML → `BlueprintDto`  
- Validate structure (required fields, dimensions, block positions)

## 1.3 Implement serializer  
- `BlueprintDto` → YAML  
- Ensure stable ordering for deterministic diffs

### Output of Phase 1  
- Ability to load and save canonical YAML blueprints  
- No UI yet  
- No derived artifacts yet

# **Phase 2 — Derivation Engine (Core of the System)**

This is the heart of the new architecture.

## 2.1 Derive **materials list**  
Count blocks by `id`.  
Group by namespace (`minecraft:`).  
Sort by frequency.

## 2.2 Derive **legend**  
Map each unique block ID to a single‑character alias.  
Rules:

- Prefer first letter of block type  
- Resolve collisions with numeric suffixes  
- Guarantee stable alias assignment

## 2.3 Derive **layer grids**  
For each Y layer:

- Create a 2D array `[z][x]`
- Fill with alias from legend
- Use `.` for air

## 2.4 Derive **phases** (unless overridden)  
Group blocks by tags:

- `foundation`
- `wall`
- `roof`
- `interior`
- `decorative`
- `redstone`
- etc.

If no tags exist, fallback to Y‑ascending grouping.

## 2.5 Derive **validation report**  
Check:

- Material count mismatches  
- Blocks outside dimensions  
- Missing blockstate for blocks that require it  
- Duplicate positions  
- Unrecognized block IDs  
- Missing tags for known semantic roles (door, bed, etc.)

## 2.6 Derive **markdown documentation**  
Generate:

- Header  
- Materials table  
- Layer grids  
- Build sequence  
- Notes (auto‑generated from tags)  
- Validation warnings  

### Output of Phase 2  
A complete derived artifact pipeline that turns canonical YAML into:

- Materials summary  
- Legend  
- Layer grids  
- Phases  
- Validation  
- Markdown docs  

This is the “compiler” for blueprints.

# **Phase 3 — Blazor Editor (Primary Human Interface)**

## 3.1 Load/save pipeline  
- Load YAML → canonical model  
- Save canonical model → YAML  
- Derived artifacts update live

## 3.2 Layer grid editor  
- Render derived grid  
- Click cell → open block detail panel  
- Editing grid updates canonical block list  
- Regenerate derived artifacts on change

## 3.3 Block detail panel  
Editable fields:

- Block ID (dropdown)  
- BlockState (dynamic key/value editor)  
- Tags (chips editor)  
- Position (locked unless moving block)

## 3.4 Block list editor  
- Tabular view of all blocks  
- Filter by tag, block ID, Y layer  
- Bulk operations (e.g., “set all roof slabs to top-half”)

## 3.5 Legend viewer  
- Read‑only (derived)  
- Shows alias → block ID mapping

## 3.6 Validation panel  
- Live warnings/errors  
- Click warning → jump to offending block

## 3.7 Optional: 3D preview  
- Voxel renderer (WebGL or Canvas)  
- Toggle layers on/off  
- Rotate camera

### Output of Phase 3  
A fully interactive blueprint editor where YAML is canonical and everything else is derived.

# **Phase 4 — Integration with MemorySmith.Agent**

## 4.1 Replace markdown parser with YAML loader  
- Remove old markdown parsing logic  
- Replace with YAML → canonical block list

## 4.2 Update `BlueprintExecutor`  
- Accept canonical `PlacementBlock` list  
- Ignore layers/materials (derived only)

## 4.3 Update HTN planner  
Use:

- `tags` for semantic grouping  
- `phases` for decomposition  
- `state` for correct placement (facing, half, etc.)

## 4.4 Update adapter (Node.js)  
Ensure blockstate is passed correctly to Mineflayer:

- Doors  
- Beds  
- Torches  
- Stairs  
- Slabs  
- Logs  
- Redstone components

### Output of Phase 4  
The agent builds from canonical YAML with full blockstate fidelity.
