MemorySmith.Agent Codebase Audit Sprint-35 MemorySmith.Agent Sprint-35 Deep Code Audit and Architectural Review Executive Summary Overview This report presents a comprehensive, evidence-based audit of the MemorySmith.Agent repository at the latest commit on the sprint-35-llm-first branch 1 .

The audit covers every line of code, all project files, and the full set of supporting artifacts, with a focus on in-progress areas, brittle conditions, error handling, architectural design, and technical debt.

The analysis is grounded in the current sprint plan, CI/test status, and the explicit goal of avoiding legacy patterns and technical debt in this greenfield project.

Key Findings Planner/Task System: The HTN planner and task library are architecturally sound, but several brittle conditions and subtle data model issues exist.

The recent fix to ReplanAsync resolves a critical exception handling bug, but context carryover and parameter preservation remain open issues.

Dependency Injection &amp; Hosted Services: DI registration and hosted service lifecycle are mostly correct, but error propagation and service startup failures can be silently swallowed, risking undetected agent downtime.

Error Handling: Several areas use patterns (e.g., Task.ContinueWith ) that can silently swallow or misroute exceptions, leading to intermittent or hard-to-debug failures.

Action Context &amp; Tool Results: There is no robust mechanism for carrying tool results between actions, limiting the planner’s ability to adapt plans based on dynamic world state or tool outputs.

Minecraft Adapter: Block ID handling is hardcoded and not version-aware, risking compatibility issues across Minecraft server versions.

Security: No evidence of prompt injection or tool-boundary taint analysis.

Testing &amp; CI: Unit test coverage is strong for planner and goal logic, but integration, mutation, and fuzz testing are not present.

CI is green, but lacks SARIF/static analysis and security gates.

Documentation &amp; Governance: The repo is self-documenting with in-repo wiki pages and ADRs, but lacks explicit CONTRIBUTING, CODEOWNERS, and SECURITY policies.

Detailed Findings and Analysis 1.

Planner and Task System 1.1.

HTN Planner ( HtnPlanner.cs ) and Task Library ( HtnTaskLibrary.cs ) The core planner implementation is architecturally sound, following the design documented in the in-repo council review and roadmap 3 .

The planner decomposes high-level goals (e.g., GatherWoodGoal ) into phase-by-phase action sequences using the task library.

The integration with the agent background service and DI is correct, and the planner is covered by 11+ unit tests.

Key Issues: ReplanAsync Exception Handling: The previous use of Task.ContinueWith with OnlyOnRanToCompletion led to intermittent silent failures.

If PlanAsync threw, the continuation would not run, and the outer try/catch would sometimes miss exceptions, returning null or propagating errors unpredictably.

This was identified as a pre-commit blocker and has been fixed in the latest commit by switching to await + try/catch 4 .

Action Context Carryover: The planner currently fires actions in sequence but does not provide a mechanism for passing tool results (e.g., the output of SearchMemory ) to subsequent actions (e.g., MoveTo ).

This limits the agent’s ability to adapt plans based on dynamic world state or tool outputs.

The council review recommends adding a context dictionary to ActionData or the planner execution context.

Parameter Preservation on Replan: When replanning a partially completed goal (e.g., gathering 5/10 logs), the planner restarts with the full target count instead of the remaining count.

This is due to the loss of the TargetCount parameter in the current replan logic.

The recommended fix is to add a method to compute the remaining count and pass it to the new goal instance.

Task Library Mutability: HtnTaskLibrary.MakeAction mutates the Arguments dictionary after init-only construction.

While this is technically safe in C# (the dictionary reference is init-only, but the contents are mutable), it is a subtle gotcha that could lead to bugs if the dictionary is replaced or shared across threads.

Recommendations: Refactor all planner async flows to use await + try/catch , never ContinueWith .

Add a mutable context dictionary to ActionData or the planner execution context for tool result carryover.

Implement parameter preservation in replanning logic for all goals with dynamic targets.

Consider making HtnTaskLibrary extensible via plugin registration in Phase 4.

Confidence: 93% (core logic is correct and testable; minor issues remain).

2.

Dependency Injection and Hosted Service Lifecycle 2.1.

AgentBackgroundService and DI Registration The agent’s main execution loop is implemented as a hosted service ( AgentBackgroundService ), registered as a singleton and added to the DI container.

The planner is injected as a required dependency, and the service is started at application startup.

Key Issues: Silent Service Startup Failures: If the planner DI registration fails, the hosted service may not start, and the failure can be silently swallowed.

This risks undetected agent downtime.

Consecutive Failure Guard: The background service implements a 3-strike failure guard, but the counter only resets on full success.

Partial progress (e.g., gathering some but not all logs) does not reset the counter, which could lead to premature agent shutdown.

Recommendations: Ensure all DI registration failures are logged and surfaced at startup.

Refine the failure guard logic to reset on partial progress, not just full success.

Add health check endpoints and structured logging for service lifecycle events.

Confidence: 87% (mostly correct, but error propagation and lifecycle edge cases need attention).

3.

Error Handling and Async Patterns 3.1.

Task Continuation and Exception Semantics The use of Task.ContinueWith with OnlyOnRanToCompletion and similar patterns is brittle and can lead to silent error swallowing or misrouted exceptions.

This is a well-known pitfall in C# async programming 4 .

Key Issues: Silent Exception Swallowing: If the antecedent task faults, the continuation may not run, and exceptions can be lost or only observed via Task.Exception , which is not always surfaced.

Intermittent Null/Exception Returns: Depending on task scheduling, callers may receive null or an exception, making error handling unpredictable.

Recommendations: Replace all uses of ContinueWith with async/await and explicit try/catch .

Ensure all exceptions are logged and surfaced to callers.

Add unit tests for all error paths, including planner and tool failures.

Confidence: 80% (recent fixes address major bugs, but subtle issues may persist).

4.

Action Context and Tool Result Carryover 4.1.

Design Pattern: Action Context The current planner design does not provide a mechanism for carrying tool results (e.g., from SearchMemory ) between actions.

This limits the agent’s ability to adapt plans based on dynamic world state or tool outputs.

Key Issues: No Context Carryover: Tool results are not injected into subsequent actions, leading to hardcoded or static plans.

Limited Adaptivity: The agent cannot adjust its behavior based on the outcome of previous tool calls.

Recommendations: Add a mutable context dictionary to ActionData or the planner execution context.

Update all tool and action implementations to read from and write to the context as needed.

Add tests for adaptive planning and context-driven action sequences.

Confidence: 85% (pattern is well-understood; implementation is pending).

5.

Minecraft Adapter: Block ID Versioning and Compatibility 5.1.

Block ID Handling The task library currently uses hardcoded block IDs (e.g., minecraft:oak_log ), which are only valid for Minecraft 1.13+ 5 .

Older server versions use different IDs (e.g., log ), risking compatibility issues.

Key Issues: Hardcoded Block IDs: Plans and actions use static block IDs, which may not match the target server version.

No Version Awareness: The adapter config does not include a Minecraft version field, making it impossible to parameterize block IDs.

Recommendations: Add a MinecraftVersion field to the adapter config.

Refactor the task library to accept block IDs as parameters, and implement version-aware translation.

Add tests for multiple server versions and block ID mappings.

Confidence: 88% (issue is well-understood; fix is straightforward).

6.

Agent Security: Prompt Injection, Taint Analysis, and OWASP Agentic Top 10 6.1.

Security Posture The codebase currently lacks explicit defenses against prompt injection, tool-boundary taint analysis, and MCP configuration risks.

No evidence of input sanitization or secret management is present.

Key Issues: Prompt Injection Risk: No input validation or prompt boundary enforcement is present, exposing the agent to goal hijacking and system prompt injection attacks 2 .

Tool-Boundary Taint Analysis: No taint tracking or input validation is implemented for tool calls, risking command injection, SQL injection, and other attacks.

MCP Configuration Risks: Secrets and credentials are not managed or scanned, and MCP server connections are not authenticated or version-pinned.

Recommendations: Integrate agent-audit in CI to scan for prompt injection, taint flows, and MCP config risks.

Add input validation and output sanitization for all tool and memory operations.

Implement secret scanning and SCA for all .NET repositories.

Add authentication and version pinning for all MCP server connections.

Confidence: 65% (no critical vulnerabilities found, but lack of defenses is risky).

No SARIF or Static Analysis Integration: CI does not upload SARIF reports or enforce static analysis/security gates.

Recommendations: Add integration tests for agent-planner-tool roundtrips.

Integrate mutation testing (Stryker.NET) and basic fuzzing for tool inputs.

Add SARIF output and upload to GitHub Security tab.

Gate merges on high-severity static analysis findings.

Confidence: 85% (unit coverage is strong; integration and mutation coverage lacking).
