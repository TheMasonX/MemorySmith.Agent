# TSK-0110: ApiKeyMiddleware Safety Hardening

**Priority:** P1 — High
**Status:** Done
**Sprint:** 46 (Wave C)
**Theme:** Tightening the Contracts
**Council:** sprint46-wavec-plan-council-2026-06-24

## Description

`ApiKeyMiddleware` currently allows all `/api/*` requests when `Agent:ApiKey` is not configured. This is fail-open by omission — a deployment accident waiting to happen. Add an explicit `AllowUnauthenticatedApi` flag (default `false`) so that auth is opt-in rather than implicit.

## Evidence

- **Source:** `WebUI.Blazor/ApiKeyMiddleware.cs`
- **Validated claims:** Deep dive audit finding 5 (95% confidence), addendum finding 6 (97% confidence), sprint-45 audit
- **Council:** Data Model Architect rated this P0-class risk

## Requirements

1. Add `Agent:AllowUnauthenticatedApi` bool setting (default `false`)
2. When `false` + missing or invalid API key → block all `/api/*` with 401
3. Emit `LogWarning` on startup when auth is disabled or key is missing
4. Keep dev convenience via explicit `AllowUnauthenticatedApi=true`

## Acceptance Criteria

- [ ] Setting `Agent:AllowUnauthenticatedApi=false` with missing `Agent:ApiKey` blocks all `/api/*` requests with 401
- [ ] Startup log contains `LogWarning` when auth is disabled
- [ ] Setting `Agent:AllowUnauthenticatedApi=true` preserves existing dev workflow
- [ ] `dotnet test` passes with zero failures
- [ ] `dotnet build` passes with zero warnings
