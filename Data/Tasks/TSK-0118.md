# TSK-0118: Resolve Chat Interpretation Split-Brain Architecture

**Priority:** P2 — Medium
**Status:** Backlog
**Sprint:** 47+
**Theme:** Architecture Consistency
**Source Audit:** `msa_sprint35-llm-first_audit_2026-06-24.md` (P0 Finding 2)

## Description

The codebase has two parallel chat interpretation paths with different semantics:
1. `LlmChatInterpreter` — early-returns on deterministic fast paths (`CreateGoal`, `CancelGoal`, `QueryHelp`, `QueryStatus`, `NavigateTo`)
2. `ChatInterpreter` — still creates `CreateGoal` interpretations directly for build/gather/craft/navigate routes

This conflicts with the Sprint 35 architecture contract: "Parsers never create goals" and `Chat → IntentDraft → Planner → Goal`. Common commands bypass the LLM entirely, and the system can't reason about ambiguous requests in a unified way.

## Evidence

- **Validated by:** Sprint-35 audit (99% confidence)
- **Sources:**
  - `Agent.Planning/LlmChatInterpreter.cs` — early-return fast paths
  - `Agent.Planning/ChatInterpreter.cs` — creates goals directly
  - Sprint 35 handoff: "Parsers never create goals"
- **Impact:** Two different interpretation styles create overlap and make debugging intent behavior hard

## Requirements

1. Keep only stop/status/inventory/help as deterministic shortcuts
2. Route everything else through semantic intent (IntentDraft)
3. Move goal creation exclusively into the planner/intent manager layer
4. Remove or retire the old `ChatInterpreter` direct-goal-creation path

## Acceptance Criteria

- [ ] Only `CancelGoal`, `QueryStatus`, `QueryInventory`, `QueryHelp` retain deterministic fast paths
- [ ] All build/gather/craft/navigate intents go through `IntentDraft → IntentManager → GoalRequest → GoalFactory`
- [ ] Old `ChatInterpreter` goal-creation path either removed or clearly marked for removal
- [ ] All existing tests pass
- [ ] `dotnet build` passes with zero warnings
