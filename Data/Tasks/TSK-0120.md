# TSK-0120: Wire ILogger into ToolDispatcher Construction

**Priority:** P3 — Low
**Status:** Backlog
**Sprint:** 49 (Wave D)
**Theme:** Observability
**Source Audit:** `memorysmith_agent_deep_audit_2026-06-25_a3be512d.md` (Finding 5, 70% confidence)

## Description

`ToolDispatcher.Register(string name, ITool tool)` has a `LogWarning` path for duplicate registration detection, but the dispatcher is constructed as `new ToolDispatcher(journal)` in `Program.cs` without passing `ILogger<ToolDispatcher>`. This means the duplicate-registration warning cannot actually emit, muting an important diagnostic signal.

## Evidence

- **Validated by:** Deep code audit (70% confidence)
- **Source:** `Agent.Tools/ToolDispatcher.cs` — optional `ILogger<ToolDispatcher>? logger` parameter, warns on overwrite
- **Source:** `WebUI.Blazor/Program.cs` — constructs `new ToolDispatcher(journal)` without logger
- **Impact:** Accidental overwrite bugs become harder to spot; alias registration collisions go undetected

## Requirements

1. Pass `sp.GetRequiredService<ILogger<ToolDispatcher>>()` when constructing the dispatcher in `Program.cs`
2. No behavioral changes needed — the logger is already wired in the constructor

## Acceptance Criteria

- [ ] `Program.cs` passes `ILogger<ToolDispatcher>` to `new ToolDispatcher(journal, logger)`
- [ ] Duplicate registration warnings now emit correctly
- [ ] `dotnet build` passes with zero warnings
- [ ] All existing tests pass
