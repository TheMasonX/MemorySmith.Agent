# TSK-0119: Decouple Emergency Stop Delivery from Queue Mutation in ClearAndEnqueueAsync

**Priority:** P2 — Medium
**Status:** Backlog
**Sprint:** 49 (Wave D)
**Theme:** Correctness
**Source Audit:** `memorysmith_agent_deep_audit_2026-06-25_a3be512d.md` (Finding 3, 82% confidence)

## Description

`ActionQueue.ClearAndEnqueueAsync` awaits the stop callback before acquiring the queue lock and clearing the plan. This preserves the "stop first, then clear" ordering that Sprint 35 P0-D established, but it also means a transient WebSocket send failure or cancellation on the stop path prevents the queue clear from happening at all. The bot may keep running stale actions precisely when it most needs to stop (e.g., during damage interrupt).

## Evidence

- **Validated by:** Deep code audit (82% confidence)
- **Source:** `Agent.Core/Models/ActionQueue.cs` — `ClearAndEnqueueAsync` awaits callback before lock
- **Source:** `AgentBackgroundService.TryInterruptOnDamageAsync` — passes `worldAdapter.SendActionAsync` as stop callback
- **Impact:** A WebSocket hiccup during emergency handling becomes a plan-safety failure; the highest-priority recovery path depends on best-effort network delivery

## Requirements

1. Keep "stop first" ordering as the primary path
2. If the stop callback fails (throws), log a warning but do NOT prevent the queue clear
3. Use try/finally to guarantee the queue clear/enqueue happens regardless of stop success
4. Add a test that simulates stop send failure and verifies the priority action still replaces stale work

## Acceptance Criteria

- [ ] Stop send failure is logged but does not block queue clear
- [ ] Queue clear and priority enqueue happen in all cases (stop succeeds, stop fails, stop not provided)
- [ ] New test verifies behavior when stop callback throws
- [ ] All existing tests pass
- [ ] `dotnet build` passes with zero warnings
