# TSK-0116: Move Creative-Mode Build Handling into Decomposer Layer

**Priority:** P2 — Medium
**Status:** Backlog
**Sprint:** 47+
**Theme:** Architecture Consistency
**Source Audit:** `msa_followup_audit-6-24-26.md` (Finding 1)

## Description

`HtnPlanner` still contains a creative-mode special case for `BuildGoal` that generates `SearchMemory` + `MoveTo` + block placement actions via `CreateCreativeBuildActions(...)`. But `PlannerRouter` prefers registered decomposers first, so `BuildGoalDecomposer` (which always delegates to `HtnTaskLibrary.DecomposeBuild` without branching on creative mode) handles production routing. The creative fast path is likely no longer exercised — a regression risk because creative mode has different execution semantics (no resource gathering).

## Evidence

- **Validated by:** Followup audit (90% confidence)
- **Sources:**
  - `Agent.Planning/HtnPlanner.cs` — creative-mode branch for `BuildGoal`
  - `Agent.Planning/Decomposition/BuildGoalDecomposer.cs` — no creative-mode awareness
  - `Agent.Planning/PlannerRouter.cs` — prefers decomposers first
- **Impact:** Creative mode may force the agent through survival-style resource gathering, wasting time or producing nonsensical plans

## Requirements

1. Move creative-mode handling into `BuildGoalDecomposer` or a shared build-plan service
2. Ensure all build routes (decomposer, fallback, direct) respect the same creative/survival semantics
3. Add a regression test that routes a `BuildGoal` through `PlannerRouter` with creative mode enabled

## Acceptance Criteria

- [ ] Creative-mode `BuildGoal` routed through `PlannerRouter` generates appropriate creative actions
- [ ] Survival-mode behavior is unchanged
- [ ] New regression test covers creative mode through router
- [ ] All existing tests pass
- [ ] `dotnet build` passes with zero warnings
