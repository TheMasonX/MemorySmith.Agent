# TSK-0115: Unify ActionQueue Synchronization

**Priority:** P2 — Medium
**Status:** Backlog
**Sprint:** 47+
**Theme:** Correctness
**Source Audit:** `msa_code_audit_report-6-24-26.md` (Finding 5), `msa_followup_audit-6-24-26.md` (Finding 3)

## Description

`ActionQueue` uses `ConcurrentQueue<ActionData>` but mixes lock-protected (`EnqueueAll`, `ClearAndEnqueue`) and lock-free (`Enqueue`, `Clear`, `Dequeue`, `Peek`) operations. The documented "atomic interrupt" guarantee of `ClearAndEnqueue` is not actually global because `Enqueue` and `Clear` don't take the same lock. This can produce out-of-order actions or stale actions surviving an interrupt under concurrent access.

## Evidence

- **Validated by:** Code audit (84% confidence), followup audit (84% confidence)
- **Source:** `Agent.Core/Models/ActionQueue.cs`
- **Impact:** Subtle race windows around priority interrupts; "atomic clear + priority action" guarantee weaker than callers may assume

## Requirements

1. Either enforce one lock for all mutating operations, or narrow the documented guarantee to best-effort local atomicity only
2. If the queue models strict priority semantics, use a single synchronization strategy consistently
3. Add a concurrency test that interleaves `Enqueue`, `Clear`, and `ClearAndEnqueue`

## Acceptance Criteria

- [ ] All mutating operations use the same lock (or docs are corrected)
- [ ] Concurrency test verifies `ClearAndEnqueue` is atomic relative to `Enqueue`/`Clear`
- [ ] All existing tests pass
- [ ] `dotnet build` passes with zero warnings
