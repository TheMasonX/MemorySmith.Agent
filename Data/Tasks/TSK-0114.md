# TSK-0114: Preserve Structured Exception Metadata in ToolDispatcher

**Priority:** P1 — High
**Status:** Done
**Sprint:** 49 (Wave E)
**Theme:** Observability
**Source Audit:** `msa_code_audit_report-6-24-26.md` (Finding 4), `msa_followup_audit-6-24-26.md` (Finding 5)

## Description

`ToolDispatcher.CallAsync` catches arbitrary exceptions and returns `ToolResult(false, ...)`, preserving only `ex.Message` in the journal. Stack traces, exception types, tool names, and correlation IDs are lost. This makes it impossible to distinguish tool bugs from expected runtime failures without additional instrumentation.

## Evidence

- **Validated by:** Code audit (93% confidence), followup audit (93% confidence), sprint-35 audit
- **Source:** `Agent.Tools/ToolDispatcher.cs` — broad `catch (Exception ex)` block
- **Impact:** Bugs inside tools become indistinguishable from legitimate runtime failures; root cause analysis is slower; silent "failure-as-data" masks defects

## Requirements

1. Keep the safe `ToolResult(false, ...)` user-facing boundary
2. Before returning, store structured exception metadata: exception type, stack trace, tool name, correlation ID
3. Use an internal "unexpected exception" journal entry type or structured log
4. Include exception type and stack trace in the journal path even if user-facing output stays sanitized

## Acceptance Criteria

- [x] `ToolResult(false, ...)` is preserved as the user-facing failure channel
- [x] Exception type and stack trace are recorded in the journal or structured log
- [x] Tool name and correlation ID are included in the metadata
- [x] New tests verify exception metadata is captured on tool failure
- [x] All existing tests pass
- [x] `dotnet build` passes with zero warnings
