# TSK-0112: Fix CraftItem Prerequisite Count Scaling

**Priority:** P1 — High
**Status:** Backlog
**Sprint:** 47+
**Theme:** Correctness
**Source Audit:** `msa_code_audit_report-6-24-26.md` (Finding 1), `msa_followup_audit-6-24-26.md` (Finding 2)

## Description

`DecomposeCraftItem` in `HtnTaskLibrary.cs` does not scale iron/stone prerequisite gathering by the requested `count`. Planks handling scales correctly (`logsNeeded = (count + PlanksPerLog - 1) / PlanksPerLog`), but iron ingot, coal, and cobblestone calculations use single-item recipe requirements only. A `CraftItemGoal("iron_pickaxe", 2)` will under-gather ore and fail on the second craft.

## Evidence

- **Validated by:** Code audit (96% confidence), followup audit (96% confidence)
- **Source:** `Agent.Planning/HtnTaskLibrary.cs` — `DecomposeCraftItem` method
- **Impact:** Multi-craft requests silently plan too little mining/smelting/cobble gathering; manifests as repeated replanning or "mystery" stalls

## Requirements

1. Multiply prerequisite needs by `count` for all count-sensitive branches (iron ingots, coal, cobblestone, stone), matching how planks already scale
2. Consider refactoring recipe logic into a small recipe model so "ingredients per craft" × "craft count" is composed once
3. Add explicit tests for `count > 1` on at least one iron tool and one stone tool

## Acceptance Criteria

- [ ] `CraftItemGoal("iron_pickaxe", 2)` mines/smelts enough for 2 pickaxes
- [ ] `CraftItemGoal("stone_sword", 3)` gathers enough cobblestone for 3 swords
- [ ] Planks behavior unchanged (already correct)
- [ ] New tests cover `count > 1` for iron tool and stone tool
- [ ] All existing tests pass
- [ ] `dotnet build` passes with zero warnings
