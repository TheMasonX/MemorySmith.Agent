# TSK-0100: WebSocketBridge ReceiveLoopAsync Structured Error Handling

**Priority:** P0 — Critical
**Status:** Backlog
**Sprint:** 46 (Wave A)
**Theme:** Observability First
**Council:** audit-validation-sprint46-plan-council-2026-06-24

## Description

The entire event receive loop in `WebSocketBridge.cs` is fire-and-forget (`_ = ReceiveLoopAsync(...)`). If it crashes, the agent goes permanently deaf with no diagnostic — no events arrive, no actions complete, no replan triggers. The agent appears alive (REST API responds, websocket connected) but is effectively brain-dead.

This is the #1 reliability risk in the entire codebase.

## Evidence

- **Source:** `Agent.World.Minecraft/WebSocketBridge.cs` line ~378
- **Validated claim:** M-2 (council finding — not called out in original audits)
- **Original audit missed this because:** Audits focused on planner/architecture; didn't trace the event receive path

## Requirements

1. Wrap `ReceiveLoopAsync` in a try-catch that logs the exception type, stack trace, and context at `LogError` level
2. Implement reconnection logic: max 3 retries with 5s delay between attempts
3. On permanent failure (all retries exhausted), trigger emergency stop via `SendEmergencyStop`
4. Ensure the restructured loop still starts correctly from `MinecraftAdapter` or whatever owns the bridge lifecycle

## Acceptance Criteria

- [ ] Inject exception into loop body → log clearly states "WebSocket receive loop crashed: {exception}"
- [ ] Reconnect attempts are logged with attempt count and delay
- [ ] After max retries, emergency stop is triggered
- [ ] Normal operation is unaffected (no behavioral change when no exception occurs)
- [ ] `dotnet test` passes with zero failures
- [ ] `dotnet build` passes with zero warnings
