# TSK-0117: Add Post-Craft/Post-Smelt Inventory Reconciliation

**Priority:** P2 — Medium
**Status:** Backlog
**Sprint:** 47+
**Theme:** Correctness
**Source Audit:** `msa_sprint35-llm-first_audit_2026-06-24.md` (P0 Finding 1)

## Description

After craft/smelt completion, `sendBotStatus()` is not reliably invoked to reconcile inventory. The world state projector applies inferred changes (`ApplyCraftComplete`/`ApplySmeltComplete`), but the actual inventory can diverge from projected state. Without a forced reconciliation snapshot, completion checks can be wrong when `IsInventoryStale` is not refreshed.

## Evidence

- **Validated by:** Sprint-35 audit (98% confidence)
- **Sources:**
  - `WorldStateProjector.cs` — applies inferred changes
  - `MineflayerAdapter/index.js` — `sendBotStatus()` not clearly invoked after craft/smelt
  - Sprint-35 council doc independently identifies the same bug class
- **Impact:** Crafted/smelted output can lag behind reality; completion checks wrong when inventory stale

## Requirements

1. Ensure `sendBotStatus()` (or equivalent reconciliation event) is sent after every craft and smelt completion in the adapter
2. In C#, when `CraftCompleteEvent` or `SmeltCompleteEvent` is received, force an inventory status refresh before marking the action complete
3. Add tests verifying inventory reconciliation after craft/smelt events

## Acceptance Criteria

- [ ] Adapter sends `sendBotStatus` after craft and smelt completion
- [ ] C# handler forces inventory refresh before completing the correlated action
- [ ] New tests verify inventory state after craft/smelt events
- [ ] All existing tests pass
- [ ] `dotnet build` passes with zero warnings
